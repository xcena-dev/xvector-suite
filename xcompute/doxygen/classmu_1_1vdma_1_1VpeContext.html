Detailed Description

Vector Processing Engine Context.

VpeContext integrates SRAM buffer allocation and vector functions into a single easy-to-use interface. It manages SRAM buffer allocation and provides synchronous vector operations (dot product, arithmetic, reduction, comparison).

This class provides automatic resource management with RAII semantics: all allocated buffers are automatically freed when the context is destroyed.

Note
Thread-safety: VpeContext uses internal locking for buffer operations. Multiple threads can safely call methods on the same context, but each thread should typically use its own VpeContext for best performance.
Buffer capacity: Each VpeContext can manage up to MAX_SLOTS (32) buffers. Attempting to allocate more will return INVALID_BUFFER.
Warning
All buffer offsets must be obtained via allocateBuffer(). Passing arbitrary values will cause undefined behavior.

Usage example:

VpeContext ctx(dimension, VpeIdStrategy::ByClusterId);
auto query = ctx.allocateBuffer();
auto target = ctx.allocateBuffer();
auto diff = ctx.allocateBuffer();
ctx.load(queryDram, query);
ctx.load(targetDram, target);
ctx.sub(target, query, diff);
float result = ctx.dot(diff, diff);
// Destructor automatically frees all buffers

◆ _reset()

static void mu::vdma::VpeContext::_reset ( )
static

[DANGEROUS] Reset the entire SRAM allocator state for a VPE

Resolves the lock address for the current thread's VPE, aligns it to 8 bytes, and clears the full 64-bit lock word (covering both VPEs in the Subsystem).

Postcondition
All buffer slots across the Subsystem are marked as free
Warning
MUST ONLY be called from a dedicated reset/init kernel. Calling this while another kernel is using VpeContext buffers will corrupt allocator state and cause undefined behavior.
This is not a normal operational API. It exists solely for host-driven initialization between kernel invocations.
